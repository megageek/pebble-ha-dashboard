# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

This repo is for a **Pebble smartwatch watchface that displays data from a linked Home Assistant instance** (e.g. sensor states, temperatures, switch/light status), plus a few always-available local channels (time, date, battery, steps). A basic scaffold exists at the repo root (`package.json`, `wscript`, `src/c/main.c`, emery-only) built around a channel/slot/template system, a Clay config page (`src/pkjs/config.js`) for choosing the active template and per-slot channel assignment, and a **bidirectional** Home Assistant WebSocket bridge (`src/pkjs/index.js`): HA ŌåÆ watch (10 remote channels) and watch ŌåÆ HA (battery, health metrics, connection status, device info) ŌĆö see "Architecture" below. Not yet built: gabbro support, and a real HA-side custom integration ŌĆö **this repo only contains the watch/phone half** of that relationship.

**`HA_INTEGRATION_SPEC.md`** is the standalone contract for that missing HA-side half ŌĆö the custom WebSocket API command name, channel numbering, payload shape, truncation limits ŌĆö written to be copied into whatever repo eventually builds the actual custom integration. **Whenever a change here touches the HA wire protocol** (command name/shape in `src/pkjs/index.js`, channel count/`HAn_VALUE`/`HAn_LABEL` keys in `src/c/main.c`/`package.json`, truncation limits), update `HA_INTEGRATION_SPEC.md` in the same commit ŌĆö it has no other way of staying accurate since the HA side can't be verified from this repo.

## Architecture: channels, slots, and templates

Three concepts are kept deliberately separate so layout and data source can vary independently:

- **Channel** ŌĆö a named data source holding a `ChannelData` value (kind/style/unit/range/label), indexed by the `ChannelIndex` enum and populated in `init_channels()`. **23 channels total**, each new group appended after whatever came before it (never interleaved) so a previously-persisted `SLOT_n_CHANNEL` value keeps meaning the same channel across app updates: `CHANNEL_TIME`, `CHANNEL_DATE`, `CHANNEL_BATTERY`, `CHANNEL_STEPS` (the original 4, local); `CHANNEL_HA_1`/`CHANNEL_HA_2`/`CHANNEL_HA_3` (**remote** ŌĆö Home Assistant pushes values into these via the pkjs WebSocket bridge, see below); then 9 more **local** channels added when watchŌåÆHA reporting was introduced ŌĆö `CHANNEL_BATTERY_CHARGING`, `CHANNEL_CONNECTED` (binary), `CHANNEL_ACTIVE_MINUTES`, `CHANNEL_DISTANCE`, `CHANNEL_ACTIVE_KCAL`, `CHANNEL_RESTING_KCAL`, `CHANNEL_SLEEP_MINUTES`, `CHANNEL_SLEEP_RESTFUL_MINUTES`, `CHANNEL_HEART_RATE` (numeric, updated by `update_health_channels()`/`update_battery()`/`battery_callback()` ŌĆö same underlying `health_service_*` calls as the outbound report, converted from seconds to minutes for on-screen readability via integer division, but **deliberately independent of the `REPORT_ENABLE_*` toggles**: those only gate what's sent to HA, not what the watch itself can display); then `CHANNEL_HA_4`ŌĆ”`CHANNEL_HA_10` (**remote**, 7 more), appended last ŌĆö after `CHANNEL_HEART_RATE`, not next to `CHANNEL_HA_1`/`_2`/`_3` ŌĆö when the number of both remote channels and configurable slots was boosted from 3/5 to 10/10 (see `NUM_HA_CHANNELS`/`NUM_CONFIGURABLE_SLOTS` in `main.c`), for the same backward-compatibility reason as the 9 local channels before them. Time and date are ordinary channels too, not special-cased widgets ŌĆö any channel can go in any slot.
- **Slot** ŌĆö a screen position from a layout template: a `GRect` plus a `SlotSizeClass` (`SLOT_SIZE_HERO`, `SLOT_SIZE_MEDIUM`, `SLOT_SIZE_SMALL`). Size class, not the channel's kind, decides how the assigned channel is drawn ŌĆö `HERO`/`MEDIUM` render a big/medium centered value with no label (`BITHAM_42_BOLD` / `GOTHIC_24`), `SMALL` renders a label+value stat row (label left, value right, a `draw_channel_bar()` progress bar instead of text when `style == CHANNEL_STYLE_BAR` and the channel has a range). The same channel renders differently depending which slot holds it.
- **Template** ŌĆö a function returning, given the window bounds, an array of `SlotSpec { rect, size_class, channel }`, where `channel` is a **default** assignment. Two exist: `build_template_0()` (hero=`CHANNEL_TIME`, medium=`CHANNEL_DATE`, 3 small rows=`CHANNEL_BATTERY`/`CHANNEL_STEPS`/`CHANNEL_HA_1`) and `build_template_1()` (hero=`CHANNEL_BATTERY` shown as `"100%"` text since HERO always renders text, 4 small rows=`CHANNEL_TIME`/`CHANNEL_DATE`/`CHANNEL_STEPS`/`CHANNEL_HA_1`), proving slots vary in size, count, and channel assignment between templates, and that the same channel (e.g. battery) renders differently ŌĆö bar vs. text-with-unit ŌĆö purely because of the slot it lands in. `s_templates[]` is the array of builder function pointers, indexed by `s_active_template` ŌĆö no longer a compile-time constant, see the Clay config section below.

Rendering: each slot is a plain `Layer` (not `TextLayer`), created with `layer_create_with_data(rect, sizeof(SlotRenderInfo))` storing `{channel_index, size_class}`. `slot_update_proc` reads that back and dispatches to `draw_channel_text_aligned()` (HERO/MEDIUM) or `draw_channel_small_row()` (SMALL). Any channel update ŌĆö a minute tick, the battery callback, a steps refresh, or eventually an HA push ŌĆö calls `mark_all_slots_dirty()` rather than tracking which slots currently show which channel; cheap with a handful of slots, and it stays correct automatically if slotŌåÆchannel assignment changes later. The separator line's position is template-agnostic too: `compute_separator_top()` finds the lowest non-`SMALL` slot across whatever slots the active template produced, rather than assuming a fixed slot index holds the header.

### Channel value model: kind, style, unit, range

Each channel's value is a small tagged struct (`ChannelData` in `src/c/main.c`), not a plain string, because channels vary in shape:

- **`kind`** ŌĆö `CHANNEL_KIND_NUMERIC`, `CHANNEL_KIND_TEXT`, or `CHANNEL_KIND_BINARY`. Determines what the raw `value`/`text` field means (a number, a short string, or 0/1).
- **`unit`** ŌĆö optional suffix for numeric channels (e.g. `"%"`), appended when rendered as text. Empty string means no unit.
- **`min`/`max`/`has_range`** ŌĆö a numeric channel may declare a range for context/validation even if it's still shown as text (e.g. a temperature with a plausible range but no bar).
- **`style`** ŌĆö `CHANNEL_STYLE_RAW` or `CHANNEL_STYLE_BAR`, an **explicit, independent choice** from whether a range exists. A ranged numeric channel still defaults to text unless `style` is set to `BAR`. This is deliberate: range and rendering style are decoupled so a future Clay config page can let the user pick "show as bar" per slot without that decision being baked into the data source. Bar rendering only ever applies when `kind == NUMERIC && has_range` (checked in `draw_channel_small_row()`); anything else falls back to text regardless of `style`, and `HERO`/`MEDIUM` slots always render as centered text since a bar wouldn't make sense there.
- Binary channels render as `"ON"`/`"OFF"` text today; a future pass can add an icon/dot style once a real binary source (e.g. an HA switch entity) exists to test against.

All bar math is integer-only (`(value - min) * width / range`), consistent with the skill's no-floating-point rule.

### Status dot group: colored indicators additive to a slot's normal content

Up to `MAX_DOTS` (4) small colored circles, shown **in addition to** whatever channel a slot already displays ŌĆö for binary-ish measures (a switch, a connection state) that don't warrant a full label+value row of their own. This is a single shared group per layout (not per-slot): `s_dot_group_slot` (-1 = disabled, else `0..NUM_CONFIGURABLE_SLOTS-1`) names which one of the 10 configurable slots also renders the dot row; `s_dot_channels[MAX_DOTS]` (-1 = empty) names which channel feeds each dot position.

- **Additive, not a replacement.** `slot_update_proc()` compares `info->slot_index == s_dot_group_slot` **at draw time** (not baked into a bool when the layer was created) ŌĆö `SlotRenderInfo.slot_index` is just each slot's own index among the 10 configurable ones, set once in `load_active_layout()` and never touched again, so changing which slot hosts the dots only needs `mark_all_slots_dirty()`, never a layout rebuild (`destroy_active_layout()`/`load_active_layout()`). When the designated slot draws, it reserves a strip on the *right* (width, not height ŌĆö Pebble's `SLOT_SIZE_SMALL` rows are only ~30px tall, too short to spare vertically, but 200px-wide screens have width to spare) sized to however many dots are actually assigned (`count_assigned_dots()`), narrowing (not replacing) `content_bounds` before the normal `draw_channel_text_aligned()`/`draw_channel_small_row()` call.
- **Any channel can be a dot** ŌĆö local (`CHANNEL_BATTERY_CHARGING`, `CHANNEL_CONNECTED`) or remote (`CHANNEL_HA_1/2/3`). `channel_is_on()` decides "on" generically by `kind`: `BINARY`/`NUMERIC` are on when non-zero, `TEXT` (the only kind HA channels use) is on only when the value is the **exact lowercase literal `"on"`** ŌĆö deliberately not case-insensitive or fuzzy, so the wire contract in `HA_INTEGRATION_SPEC.md` can just say "send `on`/`off`" without hedging.
- **Colors** come from `ChannelData.on_color`/`.off_color` (8-char name strings, e.g. `"green"`, `"red"`), resolved to a `GColor` by `color_from_name()` ŌĆö a small fixed palette (`red`/`orange`/`yellow`/`green`/`blue`/`purple`/`white`/`gray`) shared verbatim between Clay's color-picker `select` options (`config.js`'s `COLOR_OPTIONS`) and what HA sends over the wire, so there's one color vocabulary everywhere rather than two. Unrecognized/empty names fall back to gray rather than erroring.
- **`hide_when`** (`ChannelData.hide_when`: 0=none, 1=hide-when-on, 2=hide-when-off, via `parse_hide_when("none"/"on"/"off")`) lets a dot disappear entirely in the "uninteresting" state (e.g. hide a door sensor's dot unless the door is actually open) ŌĆö `draw_dot_group()` still reserves that dot's position (so other dots don't jump around when one toggles visibility), it just skips the `graphics_fill_circle()` call for it.
- **Local channels' colors/hide-when are Clay-configurable** (`CHARGE_ON_COLOR`/`CHARGE_OFF_COLOR`/`CHARGE_HIDE_WHEN`, `CONN_*` ŌĆö 6 keys, persisted individually via `persist_write_string()`/`persist_write_int()`, applied in `inbox_received_callback()` via the shared `apply_dot_color_config()` helper). **Remote (HA) channels' colors/hide-when are set by HA itself**, as optional `on_color`/`off_color`/`hide_when` fields alongside `value`/`label` in the same `pebble_channel_update` payload (see `HA_INTEGRATION_SPEC.md`) ŌĆö not Clay-configurable, since HA already owns that channel's meaning.
- **Size is configurable, one shared setting for the whole group** (not per-dot) ŌĆö `s_dot_size` (`DotSizeOption`: `SMALL`/`MEDIUM`/`LARGE`, indexing parallel `DOT_DIAMETERS`/`DOT_SPACINGS` lookup tables: `{6,9,12}`px diameter / `{5,6,8}`px spacing), persisted at `PERSIST_KEY_DOT_SIZE`, defaulting to `LARGE` ŌĆö the fixed 12px/8px that was the *only* size before this was made configurable, kept as the default so existing configs don't visually change. Both `draw_dot_group()`'s dot positions/radii and `slot_update_proc()`'s reserved-width calculation read `s_dot_size` fresh each draw, so changing it is redraw-only, same as everything else in this section.
- **Config surface**: `config.js` adds a "Status Dots" section ŌĆö `DOT_GROUP_SLOT` (which slot, `-1`..`9`), `DOT_0_CHANNEL`ŌĆ”`DOT_3_CHANNEL` (`DOT_CHANNEL_OPTIONS`, restricted to the 12 dot-capable channels + "None", not all 23 ŌĆö a numeric/text channel as a dot wouldn't mean anything), `DOT_SIZE` (Small/Medium/Large), and the 6 local color/hide-when selects. All forwarded to the watch via `WATCH_SETTINGS_KEYS` in `index.js` like the other slot/template settings.
- **Testing**: `pebble send-app-message --emulator emery --uint <DOT_GROUP_SLOT_key>=2 <DOT_0_CHANNEL_key>=7 <DOT_1_CHANNEL_key>=8` assigns dots into slot 2; add `<DOT_SIZE_key>=0` (small) / `=1` (medium) / `=2` (large) to the same command to test sizing; `--string <CHARGE_HIDE_WHEN_key>=off` exercises hide-when; `--string <HA1_VALUE_key>=on <HA1_ON_COLOR_key>=red` exercises HA-driven dot coloring (numeric keys from `build/js/message_keys.json`). Verified visually in the emulator: dots render alongside the BATT bar without disturbing it, all 3 sizes are visibly distinct, a hidden dot leaves its neighbor's position unchanged, and an HA channel's `on_color` correctly overrides the dot's color while its own slot still shows the raw text value.

### Channel colors: background, value, and label ŌĆö general-purpose, not just dots

Separate from (and independent of) the dot on/off colors above, every channel can also set a background color for whichever slot displays it, a foreground color for its value text, and a foreground color for its label text ŌĆö these apply to the channel's *normal* rendering, not just when it's in the dot group. `ChannelData.bg_color`/`.value_color`/`.label_color` (8-char name strings, same shared palette as the dot colors), all defaulting to an empty string, meaning "no override" ŌĆö `draw_channel_text_aligned()` and `draw_channel_small_row()` fall back to whatever color the slot's size class would normally use (white for HERO/SMALL values, light gray for MEDIUM values and all labels) when a field is empty, so an unconfigured channel looks exactly as it did before this option existed. `slot_update_proc()` fills the whole slot rect with `bg_color` (if set) before anything else draws, including the dot strip, so a colored slot's dots still render on top of it. `label_color` only has a visible effect in `SLOT_SIZE_SMALL`, the only size class that renders a label at all ŌĆö HERO/MEDIUM silently ignore it, same as they already ignore `style: BAR`.

- **Per-channel, like on_color/off_color, not per-slot** ŌĆö a channel's colors follow it wherever it's assigned, rather than being a property of the screen position. Reassigning a slot to a different channel changes that slot's colors along with everything else about it.
- **Local channels are Clay-configurable, one 3-color group per channel** (`config.js`'s `CHANNEL_COLOR_FIELDS` lists all 13 non-HA channels ŌĆö TIME/DATE/BATTERY/STEPS/CHARGE/CONN/ACTIVE_MINUTES/DISTANCE/ACTIVE_KCAL/RESTING_KCAL/SLEEP_MINUTES/SLEEP_RESTFUL_MINUTES/HEART_RATE ŌĆö CHARGE/CONN reusing the same messageKey prefixes as their existing dot on/off colors), each producing 3 `colorSelectWithDefault()` selects (`<PREFIX>_BG_COLOR`/`_VALUE_COLOR`/`_LABEL_COLOR`) from `COLOR_OPTIONS_WITH_DEFAULT` (the normal 8-color palette plus a leading "Default" option whose value is the empty string). Persisted at `PERSIST_KEY_LOCAL_COLOR_BASE + i*3 + f` (`i` = index into the file-scope `local_color_channels[]` array in `main.c`, `f` = 0/1/2 for bg/value/label), applied generically in `inbox_received_callback()` by looping over `local_color_channels[]` alongside three parallel non-static `MESSAGE_KEY_*` arrays (same "runtime uint32_t, can't be a static array" constraint as everywhere else these are iterated) ŌĆö one iteration handles all 39 keys rather than 13 near-duplicate blocks.
- **Remote (HA) channels get these fields directly from HA over the wire instead** ŌĆö `bg_color`/`value_color`/`label_color` are three more optional fields in the same `pebble_channel_update` payload as `on_color`/`off_color`/`hide_when` (see `HA_INTEGRATION_SPEC.md`), handled in the same per-HA-channel loop in `inbox_received_callback()`. Not persisted on the watch (same as the existing HA dot colors) ŌĆö HA is the source of truth and resends them on every reconnect via the initial subscribe result.
- **Testing**: `pebble send-app-message --emulator emery --string <STEPS_BG_COLOR_key>=purple <STEPS_VALUE_COLOR_key>=yellow <STEPS_LABEL_COLOR_key>=orange` colors a local SMALL-row channel; `--string <TIME_BG_COLOR_key>=blue <TIME_VALUE_COLOR_key>=green` exercises a HERO slot's background/value color (label_color has no visible effect there); `--string <HA1_BG_COLOR_key>=red <HA1_VALUE_COLOR_key>=orange <HA1_LABEL_COLOR_key>=white` exercises HA-driven channel styling (numeric keys from `build/js/message_keys.json`). Send empty strings back to restore defaults. Verified visually in the emulator: a SMALL row's background/label/value colors all render correctly and independently, a HERO slot's background+value color override its normal white-on-black look, and HA1's wire-supplied colors apply the same way as a local channel's Clay-configured ones ŌĆö with no regression to the default (all-empty) appearance.

## Architecture: Clay config page (template + slot assignment)

- **`src/pkjs/config.js`** is the Clay form definition: a `TEMPLATE_INDEX` select (values `0`/`1`, matching `s_templates[]`), ten `SLOT_0_CHANNEL`ŌĆ”`SLOT_9_CHANNEL` selects (values `0`ŌĆō`22`, matching all 23 `ChannelIndex` entries ŌĆö the 10 HA channels plus the 9 local health/status channels), the status-dot-group fields (`DOT_GROUP_SLOT`, `DOT_0_CHANNEL`ŌĆ”`DOT_3_CHANNEL`, 6 local color/hide-when selects ŌĆö see "Status dot group" above), the 39 general-purpose background/value/label color selects for the 13 local channels (see "Channel colors" above), 7 `REPORT_ENABLE_*` toggles (see "Per-measure reporting toggles" below), and two phone-only text inputs, `HA_URL`/`HA_TOKEN` (see the HA data flow section below ŌĆö these never reach the watch). Both built-in templates only use 5 slots, so slots 6-10 are currently always ignored on the watch side (documented as such in Clay's UI) ŌĆö the dropdown count was deliberately raised to 10 ahead of any template actually needing more, rather than needing another messageKeys/Clay pass later. Clay's conditional/dynamic-field support still isn't used; one fixed set of 10 dropdowns just covers however many slots either template (or a future one, up to 10) produces.
- **`src/pkjs/index.js`** constructs Clay with `{ autoHandleEvents: false }` and registers its own `showConfiguration`/`webviewclosed` handlers instead of Clay's default ones ŌĆö this is deliberate, not boilerplate: Clay's built-in handler would forward *every* setting (including `HA_URL`/`HA_TOKEN`) to the watch via `Pebble.sendAppMessage()`, so a `WATCH_SETTINGS_KEYS` allowlist picks out only `TEMPLATE_INDEX`/`SLOT_n_CHANNEL` before sending. `clay.getSettings()` still persists the full settings object (URL/token included) to `localStorage['clay-settings']` as a side effect, which is what the HA bridge reads from. `pebble-clay` is an npm dependency (`package.json`/`node_modules`, `package-lock.json` committed for reproducibility); webpack (already used by `pebble build` for pkjs bundling) pulls it and `config.js` into `build/pebble-js-app.js` automatically ŌĆö no wscript changes needed.
- **`package.json` `messageKeys`** are declared as a plain array (`TEMPLATE_INDEX`, `SLOT_0_CHANNEL`ŌĆ”`SLOT_9_CHANNEL`, `HA1_VALUE`/`HA1_LABEL`ŌĆ”`HA10_VALUE`/`HA10_LABEL` ŌĆö but deliberately **not** `HA_URL`/`HA_TOKEN`, which stay phone-side only); the build generates numeric key values and a `MESSAGE_KEY_*` global per key (`build/src/message_keys.auto.c` + `build/include/message_keys.auto.h`, regenerated from a clean build ŌĆö stale caches can leave the header empty, so `pebble clean` if `MESSAGE_KEY_*` symbols go missing after editing `messageKeys`). These are **runtime `uint32_t` variables, not `#define` macros** in this SDK version, so they can't be used to initialize a `static const` array ŌĆö use a plain (non-static) local array if you need to iterate them by index, as `inbox_received_callback()` does for both the slot keys and the HA channel keys.
- **C side**: `s_active_template` and `s_slot_channel_override[5]` (`-1` = "use the template's default") replace the old compile-time `ACTIVE_TEMPLATE` constant. `load_config()` reads both from persistent storage (`persist_read_int`, keys `PERSIST_KEY_TEMPLATE`/`PERSIST_KEY_SLOT_BASE+i`) on `window_load`, falling back to defaults (template 0, no overrides) on first run or out-of-range values. `inbox_received_callback()` validates and `persist_write_int()`s any `TEMPLATE_INDEX`/`SLOT_n_CHANNEL` tuples it finds, then ŌĆö if the layout actually changed ŌĆö tears down and rebuilds the slot layers immediately via `destroy_active_layout()`/`load_active_layout()`, so a config save applies live without relaunching the watchface. `app_message_register_inbox_received()` is called before `app_message_open()` in `init()`, per the standard AppMessage ordering requirement.
- **Testing without a phone**: `pebble send-app-message --emulator emery --uint <TEMPLATE_INDEX_key>=1 <SLOT_0_CHANNEL_key>=3` (numeric keys from `build/js/message_keys.json`) simulates what pkjs sends after Clay closes, without needing the actual webview ŌĆö useful since the emulator has no phone to render Clay's HTML in. **Multiple key=value pairs must go after a single `--uint` flag, not repeated `--uint` flags** ŌĆö `--uint 10000=1 --uint 10001=3` silently drops the first pair (argparse overwrite, not a merge); `--uint 10000=1 10001=3` sends both in one AppMessage. To actually render the Clay HTML (e.g. to check the form itself, not just the AppMessage it produces), stub a global `Pebble`/`localStorage` in plain Node, `require()` the built `build/pebble-js-app.js` bundle, invoke the captured `showConfiguration` listener, and grab the `data:` URI passed to the stubbed `Pebble.openURL` ŌĆö `clay.generateUrl()` throws if `Pebble`/`localStorage` aren't defined as globals, and won't run standalone otherwise.

## Architecture: Home Assistant data flow (push, custom WS command)

Home Assistant data is not reachable from watch C code directly, so the same **AppMessage + PebbleKit JS bridge** the skill uses for weather still applies structurally ŌĆö but the direction and shape of the data flow are different from the skill's default (polling) weather pattern, and it's **not** the generic HA event bus (`subscribe_events`/`fire_event`) ŌĆö that was the original design and was replaced with a custom WebSocket API command for reasons covered below. This is implemented in `src/pkjs/index.js` (the watch/phone half only ŌĆö the actual HA-side custom integration that must register the command described here doesn't exist anywhere in this repo; see `HA_INTEGRATION_SPEC.md`, the portable copy of this contract meant for that integration's own repo):

```
Home Assistant                Phone                              Watch
(custom integration     ŌåÉ Ōöé src/pkjs/index.js Ōöé ŌåÉAppMessageŌåÆ Ōöé src/c/main.c Ōöé
 registers a custom WS     Ōöé persistent WS conn Ōöé
 command)                  Ōöé                     Ōöé
        Ōöé pushes "event" msgs Ōöé relays on receipt
        ŌööŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöś
```

- **Transport is a persistent WebSocket, not REST polling** ŌĆö same connection/session HA's normal `/api/websocket` already provides, not a separate server or port. `connectToHomeAssistant()` in `src/pkjs/index.js` opens a `WebSocket` to `<HA_URL>/api/websocket`, completes HA's auth handshake (`auth_required` ŌåÆ send `{type: "auth", access_token}` ŌåÆ `auth_ok`/`auth_invalid`), then sends one custom command: `{id, type: "pebble_dashboard/subscribe_channels"}`. `HA_URL`/`HA_TOKEN` are read from `localStorage['clay-settings']` (persisted by Clay, see above) via `getHaConfig()`, never hardcoded.
- **One command both fetches current state and subscribes to future changes** ŌĆö no separate request/response round trip. The integration replies immediately with a `result` containing `{channels: [{channel, value, label?}, ...]}` for every currently-mapped channel (this is what shows real data on startup/reconnect instead of placeholders), then keeps pushing `event` messages tagged with the same request `id` whenever a channel changes, for as long as the connection stays open. Both the initial `result.channels[]` and each subsequent `event` use the identical `{channel, value, label?}` shape and are handled by the same code path (`sendHaChannelToWatch()`) ŌĆö the phone doesn't distinguish "initial" from "live" updates. Since `auth_ok` fires on every connection (the first one on `ready` and every reconnect after a drop), this one send-the-subscribe-command step covers both startup and reconnect with no separate logic needed.
- **Requires a real custom integration, not just automations** ŌĆö this is the main tradeoff of this design vs. the original generic-event-bus approach. A custom WebSocket API command (`websocket_api.async_register_command`) has to be registered in Python; it can't be wired up purely through HA automations firing events the way the old design could. The payoff: **no admin-scoped token required** (custom commands aren't admin-gated unless the integration adds that itself, unlike HA's `fire_event`), and no risk of colliding with or leaking through HA's global event bus.
- **Channels, not fixed entity fields.** Instead of hardcoding message keys per HA entity (e.g. `TEMPERATURE`), there's a fixed set of 10 numbered remote channels: `CHANNEL_HA_1`ŌĆ”`CHANNEL_HA_10` on the C side (note the enum ordering gotcha above ŌĆö `HA_4`ŌĆ”`HA_10` sit after all 16 original/local channels, not contiguous with `HA_1`/`_2`/`_3`, though `NUM_HA_CHANNELS`-indexed arrays like `ha_channels[]` in `inbox_received_callback()` just list them directly so the gap is invisible to that code), `HA1_VALUE`/`HA1_LABEL`ŌĆ”`HA10_VALUE`/`HA10_LABEL` (plus `HA1_ON_COLOR`/`HA1_OFF_COLOR`/`HA1_HIDE_WHEN`ŌĆ”, optional dot-styling ŌĆö see "Status dot group" above ŌĆö and `HA1_BG_COLOR`/`HA1_VALUE_COLOR`/`HA1_LABEL_COLOR`ŌĆ”, optional general-purpose styling ŌĆö see "Channel colors" above) as `messageKeys`. `sendHaChannelToWatch()` truncates value/label to fit the C-side buffers (`ChannelData.text[16]`/`.label[8]`) before sending, and the integration writes to whichever channel number it's configured for; the watchface doesn't know or care what entity backs a channel. **Values always arrive as strings** (`CHANNEL_KIND_TEXT`) ŌĆö there's no numeric HA channel yet, so a temperature shows as `"21.5┬░C"` text rather than a styled numeric+unit or bar value; wiring that up is a follow-on step, not yet done. (Binary display now exists via the dot group, but it's an additive overlay, not a distinct HA channel kind ŌĆö the same text-valued channel just also gets interpreted as on/off for dot purposes if it's placed in the dot group.)
- **Channel ŌåÆ slot mapping is runtime config, already built** ŌĆö see the Clay config section above.
- **Redraws are event-driven, not tick-driven, for HA data.** `inbox_received_callback()` distinguishes a *layout* change (`TEMPLATE_INDEX`/`SLOT_n_CHANNEL`, needs `destroy_active_layout()`/`load_active_layout()`) from a *value* change (`HAn_VALUE`/`HAn_LABEL`, just `mark_all_slots_dirty()`) ŌĆö an HA push never tears down and rebuilds the slot layers, it only updates the `ChannelData` in place and marks the existing layers dirty.
- **Reconnection is a phone-side concern.** The watch never holds a raw socket ŌĆö `scheduleHaReconnect()` in pkjs owns the WebSocket lifecycle with exponential backoff (starts at 2s, doubles, caps at 30s, resets to 2s on successful auth) across phone app foreground/background transitions and is the only thing that needs to handle dropped connections. Also reconnects whenever Clay config is saved (URL/token may have changed). There's no explicit unsubscribe message from the phone ŌĆö the integration should treat connection close as the subscription-cleanup signal (standard HA pattern via the `connection.subscriptions` callback).
- Still register `app_message_register_inbox_received()` **before** `app_message_open()`, and still keep `"enableMultiJS": true` in `package.json`.
- The HA base URL and long-lived access token are user-specific config, entered via the Clay config page's `HA_URL`/`HA_TOKEN` fields (see above) ŌĆö never hardcoded, never sent to the watch.
- `wscript` must use `pbl_build(..., bin_type='app')` with `js_entry_file='src/pkjs/index.js'` ŌĆö the `pbl_program` pattern does not work with pkjs.

### The other direction: watch ŌåÆ HA status reports

Separately from channel data, the watch reports its own status back out ŌĆö battery, Health metrics, connection state, device info ŌĆö via a **second** custom command, `pebble_dashboard/report_status`, over the same connection. Full field-by-field contract lives in `HA_INTEGRATION_SPEC.md`; the C-side mechanics:

- **`package.json` needs `"capabilities": ["health"]`** ŌĆö this was missing even before this feature (a real bug: `update_steps()` was already reading `HealthMetricStepCount` without it declared). Without this capability, HealthService access is not granted to the app at all, independent of the per-metric `health_service_metric_accessible()` runtime check.
- **`send_status_report()`** (in `main.c`) builds one outbound AppMessage with `app_message_outbox_begin()`/`dict_write_int32()`/`app_message_outbox_send()`, writing a `REPORT_*` key **only when that metric is actually accessible** (`write_health_metric_if_available()` checks `health_service_metric_accessible() & HealthServiceAccessibilityMaskAvailable` per metric before calling `health_service_sum()`) ŌĆö absent fields, not zeros, are how "not available on this hardware/no permission" is communicated to HA. Called from `battery_callback()` (on every charge-state change) and `tick_handler()` (every minute, alongside the clock).
- **Heart rate is instantaneous, not summed** ŌĆö `health_service_peek_current_value(HealthMetricHeartRateBPM)`, accessibility checked for "right now" (`start == end == time(NULL)`), not "today". **Deliberately does not request elevated HR sampling** (no `health_service_set_heart_rate_sample_period()` call) ŌĆö it just reads whatever the system's own default background sampling already produced, so it can be up to ~15 min stale per Pebble's docs, but costs no extra battery. Don't reintroduce elevated sampling without discussing the battery tradeoff again.
- **`send_device_info()`** sends `REPORT_MODEL`/`REPORT_FIRMWARE`/`REPORT_COLOR` (via `watch_model_name()`/`watch_color_name()` mapping the `WatchInfoModel`/`WatchInfoColor` enums to human-readable strings ŌĆö only the emery-relevant enum values are mapped, everything else falls back to `"Unknown"`) **once, in `init()`**, not repeated on every tick like the rest.
- **`REPORT_CONNECTED`** (`connection_service_peek_pebble_app_connection()`) is included for completeness but is close to tautological ŌĆö if the watch couldn't reach the phone, the report wouldn't arrive at all. Don't read much into it beyond "the message got through."
- pkjs's `appmessage` listener (`reportStatusToHomeAssistant()`) picks out every payload key starting with `REPORT_` (PebbleKit JS delivers both a numeric-key and string-key alias for each field ŌĆö the `REPORT_` prefix filter naturally ignores the numeric duplicates), strips the prefix, lowercases it, and sends the result as the `status` object in a `pebble_dashboard/report_status` command ŌĆö but **only if already authenticated to HA** (`haAuthenticated` flag, set on `auth_ok`/cleared on close); otherwise the report is silently dropped, not queued, and the next periodic report will simply try again once reconnected.

### Per-measure reporting toggles

Each report group is independently toggleable via Clay (`REPORT_ENABLE_BATTERY`/`_STEPS`/`_ACTIVITY`/`_SLEEP`/`_HEART_RATE`/`_CONNECTED`/`_DEVICE_INFO`), matching the `ReportGroup` enum (`REPORT_GROUP_BATTERY` ŌĆ” `REPORT_GROUP_DEVICE_INFO`) and `s_report_enabled[NUM_REPORT_GROUPS]` in `main.c` (all default `true`; persisted at `PERSIST_KEY_REPORT_ENABLED_BASE + i`, loaded in `load_config()`). `send_status_report()` gates every field write by its group's flag ŌĆö disabling `activity`, for instance, skips `active_seconds`/`distance_meters`/`active_kcal`/`resting_kcal` together, not one at a time.

- **Disabling a group doesn't just stop sending its fields ŌĆö it's actively reported.** `write_disabled_groups()` comma-joins the names of every currently-disabled group (`REPORT_GROUP_NAMES[]`, e.g. `"battery"`, `"heart_rate"`) into a `REPORT_DISABLED` string field, included in **every** `send_status_report()` call (not just the moment of toggling) ŌĆö this is deliberate: HA needs to tell "field absent because never accessible" apart from "field absent because the user turned it off" (the latter should remove/deactivate an entity; the former probably shouldn't), and resending the current disabled set every time means a single missed message can't leave the integration out of sync.
- **`inbox_received_callback()`** tracks these toggles as a `reporting_config_changed` flag separate from `layout_changed`/`value_changed` (a reporting toggle affects nothing about what's shown on the watch face) and calls `send_status_report()` immediately when one changes ŌĆö so a disablement reaches HA promptly rather than waiting up to a minute for the next tick.
- **Clay's `toggle` component produces a raw JS boolean** (`manipulator: 'checked'`, i.e. a checkbox's `.checked` property), not the `0`/`1` int the C side and `dict_find(...)->value->int32` expect. `webviewclosed` in `index.js` coerces any boolean value to `1`/`0` before building `watchSettings` ŌĆö this is a real, tested gotcha (see `test/ha_bridge.test.js`'s boolean-conversion assertion), not speculative.

- **`test/ha_bridge.test.js`** ŌĆö a persisted mock-WebSocket/mock-`Pebble` harness for the pkjs bridge (run `node test/ha_bridge.test.js` after `pebble build`, since it `require()`s the built `build/pebble-js-app.js`; it's deliberately **outside** `src/pkjs/` so `wscript`'s `src/pkjs/**/*.js` glob never bundles test code into the actual watchface). It covers both directions: HAŌåÆwatch (startup sequencing, initial state via `result`, a rejected/unknown command, channels 1-3 relaying independently via `event` pushes, value-only updates, value/label truncation, `on_color`/`off_color`/`hide_when` relaying and each being independently optional, `bg_color`/`value_color`/`label_color` relaying independently of the dot styling fields, out-of-range channels ignored, the boosted 4-10 channel range relaying via the same generic key-building, reconnect backoff, `webviewclosed` filtering+reconnecting and forwarding dot-group config including `DOT_SIZE`) and watchŌåÆHA (`report_status` field naming/lowercasing, dropped when unauthenticated, nothing sent for an empty status, Clay booleanŌåÆint conversion, a string-valued field relaying alongside numeric ones). **This test previously caught a real shipped bug**: `clay.getSettings(e.response)` (default `convert=true`) returns settings keyed by *numeric* AppMessage IDs (e.g. `"10000"`), not the string messageKey names (`"TEMPLATE_INDEX"`) the `WATCH_SETTINGS_KEYS` filter was looking up ŌĆö so config saves were silently forwarding an empty dict to the watch. Fixed by passing `convert=false` to get raw string-keyed settings; `Pebble.sendAppMessage()` resolves string keys against `package.json`'s `messageKeys` itself; see `src/pkjs/index.js`'s `webviewclosed` handler.
- **Testing without a real Home Assistant instance**: `pebble send-app-message --emulator emery --string <HA1_VALUE_key>="21.5┬░C" <HA1_LABEL_key>="Temp"` (numeric keys from `build/js/message_keys.json`) simulates what pkjs sends when an HA event arrives on the watch side, without needing a real HA server. To exercise the watchŌåÆHA direction for real (not just the pkjs relay logic in the mock harness), trigger a battery-state change in the emulator ŌĆö `pebble emu-battery --emulator emery --percent 65` ŌĆö which fires `battery_callback()` ŌåÆ `send_status_report()` on the actual running watch, and check `pebble logs --emulator emery` for the resulting `appmessage` payload on the phone side. To test a reporting toggle for real, `pebble send-app-message --emulator emery --uint <REPORT_ENABLE_HEART_RATE_key>=0` (numeric keys from `build/js/message_keys.json`) simulates a Clay config save disabling that group ŌĆö confirmed via logs to trigger an immediate report with `REPORT_DISABLED` set, persisted across subsequent reports. For the pkjs-side connect/auth/subscribe/relay logic itself, use `test/ha_bridge.test.js` above rather than writing new one-off scripts.
