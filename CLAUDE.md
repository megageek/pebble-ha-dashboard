# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

This repo is for a **Pebble smartwatch watchface that displays data from a linked Home Assistant instance** (e.g. sensor states, temperatures, switch/light status), plus a few always-available local channels (time, date, battery, steps). A basic scaffold exists at the repo root (`package.json`, `wscript`, `src/c/main.c`, emery-only) built around a channel/slot/template system, a Clay config page (`src/pkjs/config.js`) for choosing the active template and per-slot channel assignment, and a Home Assistant WebSocket bridge (`src/pkjs/index.js`) that relays 3 remote channels to the watch — see "Architecture" below. Not yet built: gabbro support, and a real HA-side custom integration/automation (the pkjs bridge expects a `pebble_channel_update` event with `{channel, value, label}` — nothing on the HA side fires that event yet, this repo only contains the watch/phone half).

## Building the watchface

Use the **pebble-watchface** skill (`.claude/skills/pebble-watchface/SKILL.md`, mirrored at `.agents/skills/pebble-watchface/`) to generate, build, and test this project. It defines the full research → design → implement → build → test → deliver flow and should be treated as the primary spec for how work in this repo proceeds. Key points not to rediscover from scratch:

- **Default/exclusive target platform is `emery`** (Pebble Time 2, 200x228, rectangular, 64-color). Round platforms (gabbro) are a second pass only if requested.
- **Battery rule: `tick_timer_service_subscribe(MINUTE_UNIT, ...)` only** — never `SECOND_UNIT` unless explicitly requested.
- **No floating point** — use `sin_lookup`/`cos_lookup`/`DEG_TO_TRIGANGLE`.
- Use `layer_get_bounds()` for screen dimensions; never hardcode 200x228 or other platform sizes.

## Architecture: channels, slots, and templates

Three concepts are kept deliberately separate so layout and data source can vary independently:

- **Channel** — a named data source holding a `ChannelData` value (kind/style/unit/range/label), indexed by the `ChannelIndex` enum and populated in `init_channels()`: `CHANNEL_TIME`, `CHANNEL_DATE`, `CHANNEL_BATTERY`, `CHANNEL_STEPS` — all **local**, read straight off on-watch services (`battery_state_service_peek()`, `health_service_sum_today(HealthMetricStepCount)` gated by `health_service_metric_accessible()`), no phone/network round-trip — and `CHANNEL_HA_1`/`CHANNEL_HA_2`/`CHANNEL_HA_3`, the three **remote** channels Home Assistant pushes values into via the pkjs WebSocket bridge (see below). Time and date are ordinary channels, not special-cased widgets, so they can be placed in any slot like any other channel.
- **Slot** — a screen position from a layout template: a `GRect` plus a `SlotSizeClass` (`SLOT_SIZE_HERO`, `SLOT_SIZE_MEDIUM`, `SLOT_SIZE_SMALL`). Size class, not the channel's kind, decides how the assigned channel is drawn — `HERO`/`MEDIUM` render a big/medium centered value with no label (`BITHAM_42_BOLD` / `GOTHIC_24`), `SMALL` renders a label+value stat row (label left, value right, a `draw_channel_bar()` progress bar instead of text when `style == CHANNEL_STYLE_BAR` and the channel has a range). The same channel renders differently depending which slot holds it.
- **Template** — a function returning, given the window bounds, an array of `SlotSpec { rect, size_class, channel }`, where `channel` is a **default** assignment. Two exist: `build_template_0()` (hero=`CHANNEL_TIME`, medium=`CHANNEL_DATE`, 3 small rows=`CHANNEL_BATTERY`/`CHANNEL_STEPS`/`CHANNEL_HA_1`) and `build_template_1()` (hero=`CHANNEL_BATTERY` shown as `"100%"` text since HERO always renders text, 4 small rows=`CHANNEL_TIME`/`CHANNEL_DATE`/`CHANNEL_STEPS`/`CHANNEL_HA_1`), proving slots vary in size, count, and channel assignment between templates, and that the same channel (e.g. battery) renders differently — bar vs. text-with-unit — purely because of the slot it lands in. `s_templates[]` is the array of builder function pointers, indexed by `s_active_template` — no longer a compile-time constant, see the Clay config section below.

Rendering: each slot is a plain `Layer` (not `TextLayer`), created with `layer_create_with_data(rect, sizeof(SlotRenderInfo))` storing `{channel_index, size_class}`. `slot_update_proc` reads that back and dispatches to `draw_channel_text_aligned()` (HERO/MEDIUM) or `draw_channel_small_row()` (SMALL). Any channel update — a minute tick, the battery callback, a steps refresh, or eventually an HA push — calls `mark_all_slots_dirty()` rather than tracking which slots currently show which channel; cheap with a handful of slots, and it stays correct automatically if slot→channel assignment changes later. The separator line's position is template-agnostic too: `compute_separator_top()` finds the lowest non-`SMALL` slot across whatever slots the active template produced, rather than assuming a fixed slot index holds the header.

### Channel value model: kind, style, unit, range

Each channel's value is a small tagged struct (`ChannelData` in `src/c/main.c`), not a plain string, because channels vary in shape:

- **`kind`** — `CHANNEL_KIND_NUMERIC`, `CHANNEL_KIND_TEXT`, or `CHANNEL_KIND_BINARY`. Determines what the raw `value`/`text` field means (a number, a short string, or 0/1).
- **`unit`** — optional suffix for numeric channels (e.g. `"%"`), appended when rendered as text. Empty string means no unit.
- **`min`/`max`/`has_range`** — a numeric channel may declare a range for context/validation even if it's still shown as text (e.g. a temperature with a plausible range but no bar).
- **`style`** — `CHANNEL_STYLE_RAW` or `CHANNEL_STYLE_BAR`, an **explicit, independent choice** from whether a range exists. A ranged numeric channel still defaults to text unless `style` is set to `BAR`. This is deliberate: range and rendering style are decoupled so a future Clay config page can let the user pick "show as bar" per slot without that decision being baked into the data source. Bar rendering only ever applies when `kind == NUMERIC && has_range` (checked in `draw_channel_small_row()`); anything else falls back to text regardless of `style`, and `HERO`/`MEDIUM` slots always render as centered text since a bar wouldn't make sense there.
- Binary channels render as `"ON"`/`"OFF"` text today; a future pass can add an icon/dot style once a real binary source (e.g. an HA switch entity) exists to test against.

All bar math is integer-only (`(value - min) * width / range`), consistent with the skill's no-floating-point rule.

## Architecture: Clay config page (template + slot assignment)

- **`src/pkjs/config.js`** is the Clay form definition: a `TEMPLATE_INDEX` select (values `0`/`1`, matching `s_templates[]`), five `SLOT_0_CHANNEL`…`SLOT_4_CHANNEL` selects (values `0`–`6`, matching `ChannelIndex` — includes the 3 HA channels), and two phone-only text inputs, `HA_URL`/`HA_TOKEN` (see the HA data flow section below — these never reach the watch). Both templates happen to have exactly 5 slots, so one fixed set of 5 dropdowns covers either template without needing Clay's conditional/dynamic-field support — slots beyond what the active template uses are simply ignored on the watch side.
- **`src/pkjs/index.js`** constructs Clay with `{ autoHandleEvents: false }` and registers its own `showConfiguration`/`webviewclosed` handlers instead of Clay's default ones — this is deliberate, not boilerplate: Clay's built-in handler would forward *every* setting (including `HA_URL`/`HA_TOKEN`) to the watch via `Pebble.sendAppMessage()`, so a `WATCH_SETTINGS_KEYS` allowlist picks out only `TEMPLATE_INDEX`/`SLOT_n_CHANNEL` before sending. `clay.getSettings()` still persists the full settings object (URL/token included) to `localStorage['clay-settings']` as a side effect, which is what the HA bridge reads from. `pebble-clay` is an npm dependency (`package.json`/`node_modules`, `package-lock.json` committed for reproducibility); webpack (already used by `pebble build` for pkjs bundling) pulls it and `config.js` into `build/pebble-js-app.js` automatically — no wscript changes needed.
- **`package.json` `messageKeys`** are declared as a plain array (`TEMPLATE_INDEX`, `SLOT_0_CHANNEL`…`SLOT_4_CHANNEL`, `HA1_VALUE`/`HA1_LABEL`…`HA3_VALUE`/`HA3_LABEL` — but deliberately **not** `HA_URL`/`HA_TOKEN`, which stay phone-side only); the build generates numeric key values and a `MESSAGE_KEY_*` global per key (`build/src/message_keys.auto.c` + `build/include/message_keys.auto.h`, regenerated from a clean build — stale caches can leave the header empty, so `pebble clean` if `MESSAGE_KEY_*` symbols go missing after editing `messageKeys`). These are **runtime `uint32_t` variables, not `#define` macros** in this SDK version, so they can't be used to initialize a `static const` array — use a plain (non-static) local array if you need to iterate them by index, as `inbox_received_callback()` does for both the slot keys and the HA channel keys.
- **C side**: `s_active_template` and `s_slot_channel_override[5]` (`-1` = "use the template's default") replace the old compile-time `ACTIVE_TEMPLATE` constant. `load_config()` reads both from persistent storage (`persist_read_int`, keys `PERSIST_KEY_TEMPLATE`/`PERSIST_KEY_SLOT_BASE+i`) on `window_load`, falling back to defaults (template 0, no overrides) on first run or out-of-range values. `inbox_received_callback()` validates and `persist_write_int()`s any `TEMPLATE_INDEX`/`SLOT_n_CHANNEL` tuples it finds, then — if the layout actually changed — tears down and rebuilds the slot layers immediately via `destroy_active_layout()`/`load_active_layout()`, so a config save applies live without relaunching the watchface. `app_message_register_inbox_received()` is called before `app_message_open()` in `init()`, per the standard AppMessage ordering requirement.
- **Testing without a phone**: `pebble send-app-message --emulator emery --uint <TEMPLATE_INDEX_key>=1 <SLOT_0_CHANNEL_key>=3` (numeric keys from `build/js/message_keys.json`) simulates what pkjs sends after Clay closes, without needing the actual webview — useful since the emulator has no phone to render Clay's HTML in. **Multiple key=value pairs must go after a single `--uint` flag, not repeated `--uint` flags** — `--uint 10000=1 --uint 10001=3` silently drops the first pair (argparse overwrite, not a merge); `--uint 10000=1 10001=3` sends both in one AppMessage. To actually render the Clay HTML (e.g. to check the form itself, not just the AppMessage it produces), stub a global `Pebble`/`localStorage` in plain Node, `require()` the built `build/pebble-js-app.js` bundle, invoke the captured `showConfiguration` listener, and grab the `data:` URI passed to the stubbed `Pebble.openURL` — `clay.generateUrl()` throws if `Pebble`/`localStorage` aren't defined as globals, and won't run standalone otherwise.

## Architecture: Home Assistant data flow (push, channel-based)

Home Assistant data is not reachable from watch C code directly, so the same **AppMessage + PebbleKit JS bridge** the skill uses for weather still applies structurally — but the direction and shape of the data flow are different from the skill's default (polling) weather pattern. This is now implemented in `src/pkjs/index.js` (the watch/phone half only — nothing on the actual Home Assistant side fires the expected event yet):

```
Home Assistant          Phone                              Watch
(custom integration/  ← │ src/pkjs/index.js │ ←AppMessage→ │ src/c/main.c │
 automation action)     │ persistent WS conn │
        │ fires event        │ relays on receipt
        └────────────────────┘
```

- **Transport is a persistent WebSocket, not REST polling.** `connectToHomeAssistant()` in `src/pkjs/index.js` opens a `WebSocket` to `<HA_URL>/api/websocket`, completes HA's auth handshake (`auth_required` → send `{type: "auth", access_token}` → `auth_ok`/`auth_invalid`), then sends `{type: "subscribe_events", event_type: "pebble_channel_update"}`. When that event fires HA-side, pkjs relays it to the watch **immediately** via `Pebble.sendAppMessage()` — there is no `REQUEST_*` round-trip originating from the watch, and no fixed refresh interval to reproduce from the weather template. `HA_URL`/`HA_TOKEN` are read from `localStorage['clay-settings']` (persisted by Clay, see above) via `getHaConfig()`, never hardcoded.
- **Initial state on startup/reconnect, not just wait-for-next-push.** Right after `subscribe_events` in the `auth_ok` handler, pkjs also sends `{type: "fire_event", event_type: "pebble_request_state"}`. This isn't a bespoke request/response primitive — it reuses the same `pebble_channel_update` event the HA-side automation already fires on organic changes; the automation should *also* trigger on `pebble_request_state` and re-publish current values for every channel it manages. Since `auth_ok` fires on every connection (both the initial one on `ready` and every reconnect after a drop), this one code path covers "give me current state" in both cases — no separate startup-vs-reconnect logic needed, and no change on the C side, since a state-request response arrives as an ordinary `pebble_channel_update` event indistinguishable from a live push. **Requires an admin-scoped long-lived token** — HA's `fire_event` WebSocket command is admin-only, unlike `subscribe_events`/`get_states`.
- **Channels, not fixed entity fields.** Instead of hardcoding message keys per HA entity (e.g. `TEMPERATURE`), there's a small fixed set of 3 numbered remote channels: `CHANNEL_HA_1`/`CHANNEL_HA_2`/`CHANNEL_HA_3` on the C side, `HA1_VALUE`/`HA1_LABEL`…`HA3_VALUE`/`HA3_LABEL` as `messageKeys`. The event payload is expected as `{channel: 1-3, value: "...", label: "..."}` — `sendHaChannelToWatch()` truncates value/label to fit the C-side buffers (`ChannelData.text[16]`/`.label[8]`) before sending, and a real integration/automation writes to whichever channel number it's configured for; the watchface doesn't know or care what entity backs a channel. **Values always arrive as strings** (`CHANNEL_KIND_TEXT`) — there's no numeric/binary HA channel yet, so a temperature shows as `"21.5°C"` text rather than a styled numeric+unit or bar value; wiring that up is a follow-on step, not yet done.
- **Channel → slot mapping is runtime config, already built** — see the Clay config section above.
- **Redraws are event-driven, not tick-driven, for HA data.** `inbox_received_callback()` distinguishes a *layout* change (`TEMPLATE_INDEX`/`SLOT_n_CHANNEL`, needs `destroy_active_layout()`/`load_active_layout()`) from a *value* change (`HAn_VALUE`/`HAn_LABEL`, just `mark_all_slots_dirty()`) — an HA push never tears down and rebuilds the slot layers, it only updates the `ChannelData` in place and marks the existing layers dirty.
- **Reconnection is a phone-side concern.** The watch never holds a raw socket — `scheduleHaReconnect()` in pkjs owns the WebSocket lifecycle with exponential backoff (starts at 2s, doubles, caps at 30s, resets to 2s on successful auth) across phone app foreground/background transitions and is the only thing that needs to handle dropped connections. Also reconnects whenever Clay config is saved (URL/token may have changed).
- Still register `app_message_register_inbox_received()` **before** `app_message_open()`, and still keep `"enableMultiJS": true` in `package.json`.
- The HA base URL and long-lived access token are user-specific config, entered via the Clay config page's `HA_URL`/`HA_TOKEN` fields (see above) — never hardcoded, never sent to the watch.
- `wscript` must use `pbl_build(..., bin_type='app')` with `js_entry_file='src/pkjs/index.js'` — the `pbl_program` pattern does not work with pkjs.
- **Testing without a real Home Assistant instance**: `pebble send-app-message --emulator emery --string <HA1_VALUE_key>="21.5°C" <HA1_LABEL_key>="Temp"` (numeric keys from `build/js/message_keys.json`) simulates what pkjs sends when an HA event arrives, without needing a real HA server or the WebSocket handshake. To verify the pkjs-side connect/auth/subscribe/request-state *sequencing* itself (not just its end effect on the watch), stub `Pebble`/`localStorage`/`WebSocket` as globals in plain Node, `require()` the built `build/pebble-js-app.js`, fire the `ready` listener, and script the mock socket's `onmessage` through `auth_required` → `auth_ok` → a fake `pebble_channel_update` event, asserting on what got `.send()` and what reached the stubbed `Pebble.sendAppMessage()` — this catches ordering/shape bugs that pure `send-app-message` testing can't, without needing pypkjs or a real HA server.
